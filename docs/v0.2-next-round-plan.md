# DreamAxis v0.2 Next Round Plan

## Goal

Turn DreamAxis chat from a validated repo-copilot prototype into a **more dependable verification and troubleshooting workspace** for solo developers.

This round starts from a green baseline:

- `docs/chat-acceptance-report-v0.2.md` -> `8/8` passed
- `docs/chat-acceptance-report-nvidia.md` -> `9/9` passed

The next step is **not** broader agent autonomy. The next step is to make the current chat-first execution lane sharper, safer, and more useful in day-to-day repo work.

## Product focus

Primary shape for this round:

- **chat-first**
- **verify / troubleshoot first**
- **proposal-only for edits**
- **evidence before prose**

DreamAxis should feel closer to an operator console than a generic assistant.

## What we already proved

DreamAxis can now:

- call a real external model through an OpenAI-compatible provider
- route chat into explicit modes:
  - `understand`
  - `inspect`
  - `verify`
  - `propose_fix`
- create execution bundles with child runtime executions
- return visible evidence in chat
- preserve proposal-only safety for fix suggestions
- show parent/child execution linkage in runtime

## Main problems to solve next

### 1. Verification quality still needs to be more repo-aware

Acceptance passed, but some verification probes still degrade into generic `npm run lint --if-present` or `npm run build --if-present` calls.

Next improvements:

- inspect `package.json`, `pnpm-lock.yaml`, workspace layout, and Python markers before choosing probes
- prefer the correct package manager and script entrypoint
- distinguish:
  - command missing
  - dependency missing
  - script missing
  - repo not ready
- surface those as operator-facing states, not just stderr blobs

### 2. Troubleshooting needs stronger failure summarization

Next improvements:

- extract stderr highlights automatically
- classify failure surfaces:
  - dependency/setup issue
  - missing runtime/tooling
  - code/config issue
  - browser/UI failure
- make next-step suggestions more specific to the observed failure class

### 3. Proposal mode should become more actionable

It now stays safe and grounded, but still needs better repair framing.

Next improvements:

- stronger proposal targets
- clearer candidate files
- explicit suggested command sequence
- better patch-summary language
- proposal cards in chat that are easier to scan and copy

### 4. Runtime evidence UX still needs compression

The evidence is there, but long turns can get visually heavy.

Next improvements:

- tighter inline execution cards
- collapsible raw payloads by default
- better artifact preview hierarchy
- top-level “why this failed” summary in runtime detail

## Workstreams

## Workstream A — Repo-aware verify routing

Ship:

- package-manager-aware command selection
- Node vs Python vs mixed-workspace probe selection
- better workspace marker detection before running checks
- clearer “not runnable here yet” states

Success looks like:

- verify mode chooses the right probe with less noise
- fewer false-negative build/lint failures caused by bad defaults

## Workstream B — Troubleshooting summarizer

Ship:

- stderr extraction and highlighting
- failure classification
- clearer causal summary in `What was found`
- better recommended-next-action generation

Success looks like:

- a developer can skim one turn and understand:
  - what failed
  - why it likely failed
  - what to do next

## Workstream C — Proposal UX refinement

Ship:

- richer `proposal.targets`
- `proposal.suggested_commands` that are more concrete
- explicit “not applied” visual treatment
- optional diff-outline preview text for likely file edits

Success looks like:

- `propose_fix` gives a repair plan that is actually usable without ever mutating files

## Workstream D — Acceptance + regression system

Ship:

- versioned scenario pack for:
  - DreamAxis
  - one Node repo
  - one Python repo
- repeatable local validation command
- release-gate checks for:
  - mode visibility
  - execution bundle presence
  - evidence items
  - proposal-only safety

Success looks like:

- new changes do not silently regress repo-copilot usefulness

## Suggested implementation order

1. **Repo-aware verify routing**
2. **Troubleshooting summarizer**
3. **Proposal UX refinement**
4. **Acceptance automation hardening**
5. **Runtime/chat visual compression**

## Non-goals for this round

Do **not** expand into:

- automatic file writes
- autonomous multi-agent workflows
- hidden orchestration DAGs
- broad SaaS collaboration features
- native multi-provider protocol sprawl

## Release gate for the next checkpoint

Before calling the next checkpoint successful, DreamAxis should be able to:

1. inspect a repo and choose a better verification path automatically
2. explain a failure with a short grounded summary
3. produce a proposal-only repair plan with concrete targets
4. show all relevant runtime evidence in chat and runtime
5. pass the multi-repo scenario runbook again
