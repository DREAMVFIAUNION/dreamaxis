# DreamAxis v0.3 Desktop Operator Plan

## Summary

DreamAxis v0.3 shifts the main product lane from repo-copilot-first toward a **local-first desktop computer-use agent** while preserving the constraints that already make the product trustworthy:

- local-first
- no-signup by default
- runtime-backed evidence
- gated actions
- proposal-only for code repair and file changes

This is **not** a move toward black-box autonomy. The goal is an operator-grade desktop lane that can inspect, verify, and perform confirmed Windows actions with a visible audit trail.

## Product stance

Primary direction for v0.3:

- **desktop agent primary**
- **Windows app control first**
- **gated actions**
- **runtime-backed audit**

Repo copilot remains important, but it becomes one professional lane inside a broader desktop operator product.

## Interaction contract

The chat contract should remain execution-first:

1. intent / plan
2. grounded target
3. requested desktop actions
4. approval gate
5. what ran
6. what was found
7. reflection
8. recommended next step

The key rule remains the same:

> no meaningful action without evidence, and no risky state change without confirmation

## Execution policy

### Auto-allowed by default

- system and device information reads
- process enumeration
- top-level window enumeration
- current focus window inspection
- screenshot capture
- OCR / extract-text / accessibility-tree style reads
- runtime and host capability inspection

### Explicit confirmation required

- launching an allowed app
- switching focus to another window
- click / type / hotkey actions
- navigation inside desktop apps
- download / install / open external programs
- system setting changes

### Blocked by default

- unbounded autonomous loops
- destructive system changes
- silent state-changing actions
- unrestricted execution outside allowed host and app scope

## Capability layers

### 1. Desktop grounding layer v1

Add a new grounding lane beside repo and browser grounding:

- OS / device / monitor details
- process list
- top-level windows
- active window
- screenshot summary
- OCR summary
- requested app / window / task targets from the prompt

Suggested shared concepts:

- `DesktopGroundingSignal`
- `DesktopGroundedTarget`
- `DesktopActionGuard`

### 2. Desktop runtime lane v1

Add a Windows-first runtime host beside CLI and Browser Runtime.

Initial supported actions:

- list windows
- focus window
- launch allowed app
- capture screen
- OCR / extract text
- click target
- type text
- press hotkeys
- read process / device / system info

Out of scope for v0.3 alpha:

- long autonomous loops
- unconstrained system administration
- persistent desktop memory systems
- full cross-platform parity

### 3. Chat routing upgrade

Extend the visible chat modes so desktop work is explicit:

- `understand_repo`
- `inspect_repo`
- `verify_repo`
- `propose_fix`
- `inspect_desktop`
- `operate_desktop`
- `verify_desktop`

Mode detection can stay automatic by default, but the selected mode must remain visible and user-switchable in the UI.

### 4. Approval and audit

Every state-changing desktop action should create a visible approval contract:

- action summary
- target app / window
- exact planned steps
- risk notes
- confirm / deny
- confirmed execution bundle
- runtime artifact trail

`/runtime` remains the audit surface rather than a hidden implementation detail.

## API and type direction

The goal is to extend current interfaces rather than create a separate protocol.

Primary endpoints:

- `POST /api/v1/messages`
- `POST /api/v1/messages/stream`
- `GET /api/v1/runtime-executions/{id}`
- `GET /api/v1/runtime-hosts`
- `GET /api/v1/skills`

Suggested shared types:

- `DesktopActionRequest`
- `DesktopActionStep`
- `DesktopActionApproval`
- `DesktopGroundingSignal`
- `DesktopGroundedTarget`
- `DesktopExecutionArtifact`

## Suggested implementation order

1. desktop grounding v1
2. desktop runtime host v1 for Windows
3. chat approval contract
4. Windows app control scenarios
5. README and product positioning refresh after alpha works

## Acceptance targets

Desktop operator alpha should at least pass:

- list and identify top-level Windows windows
- identify current device / system / process state
- launch one allowed app through an explicit gate
- focus a target window and capture screenshot / OCR
- execute one low-risk confirmed UI action
- return grounded target, approval, execution, evidence, and next step in chat
- preserve runtime audit visibility for all desktop actions

## Non-goals

v0.3 should **not** include:

- autonomous infinite-loop computer control
- silent file mutation
- automatic code editing
- unrestricted high-risk system administration
- macOS / Linux parity before Windows proves out
